# 技术决策记录

本文记录当前已经采用的技术和下一阶段建议，不代表后续功能已经实现。

## Vue 3 + Vite

**状态：已采用。**

选择原因：

- Vue 3 组件和 Composition API 适合把上传、任务状态、结果展示等能力拆成清晰模块。
- Vite 提供快速开发服务器、Vue 单文件组件支持和生产构建流程。
- Vite 开发代理可以让前端使用相对 `/api` 地址访问 Flask，减少开发环境地址散落。
- 该组合资料充足，适合个人作品集和渐进式开发。

当前边界：只实现 Dashboard、About 与健康检查，没有实现分析业务页面。

## TypeScript

**状态：已采用。**

选择原因：

- 为 API 响应、任务状态和未来模型结果建立明确数据结构。
- 在构建阶段提前发现字段名、空值和组件调用错误。
- 比纯 JavaScript 多一层静态检查，但不改变浏览器最终运行 JavaScript 的事实。

当前构建使用 `tsc --noEmit` 与 `vue-tsc --noEmit` 进行类型检查，避免把生成的 JavaScript 副本提交到源码目录。

## Flask

**状态：已采用。**

选择原因：

- 与 Python、PyTorch 和图像处理生态直接衔接。
- 核心概念少，适合清晰展示路由、服务层和模型适配器的调用链。
- 应用工厂与 Blueprint 足以支撑当前个人项目，无需提前拆分微服务。

当前边界：只提供健康检查 API，数据库和模型服务尚未接入。

## 先 SQLite，后 MySQL

**状态：建议用于阶段 2，尚未实现。**

原因：

- SQLite 不需要独立数据库服务，能让第一版任务管理更容易运行和测试。
- 个人 MVP 的核心风险是任务流程和模型接入，不是数据库运维。
- 后续通过 SQLAlchemy 和迁移工具约束数据库访问，可降低切换 MySQL 的业务代码改动。
- MySQL 应在任务模型稳定后用于验证连接配置、字符集、JSON、索引和迁移兼容性。

切换时需要重点验证 UUID 表示、UTC 时间、JSON 字段、布尔值、外键和 `utf8mb4` 字符集。

## 当前不引入 Redis 与 Celery

**状态：明确暂缓。**

原因：

- 当前唯一 API 是快速返回的健康检查，不存在长时任务或并发队列需求。
- 同步任务状态流尚未建立，提前加入消息队列会增加服务数量、错误路径和调试成本。
- 后续只有在模型推理明显超出普通 HTTP 请求时间、需要并发控制或任务恢复时，才重新评估异步队列。

## 当前不引入 Docker

**状态：明确暂缓。**

原因：

- 当前优先验证 Windows 本地开发和 CPU 可运行路径。
- PyTorch、模型权重、GPU 与 Windows 文件挂载会显著增加容器配置复杂度。
- 在依赖、数据库和启动方式尚未稳定前维护镜像收益较低。
- Docker 适合作为功能稳定后的可选部署与复现手段，而不是阶段 1 的前置条件。

## 当前不采用微服务

**状态：明确暂缓。**

当前规模使用单个 Vue 前端和单个 Flask 后端即可清晰表达系统边界。模型适配器可以先在后端进程内保持模块化；只有在独立扩缩容、故障隔离或多模型资源调度成为真实需求时，再考虑拆分服务。
